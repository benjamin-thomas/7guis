1 · Effects as data

In idiomatic Elm/Flame, update returns (model, Cmd) where Cmd is an opaque effect the runtime owns — you can't inspect it, so you can't unit-test which effects an action requested. We flip that: update is a pure function returning the next model plus a list of effects as plain data. 

-- An Instr is an "application effect represented as data": a description of a
-- side effect, carried out later by the interpreter. It performs nothing itself.
data Instr
  = StartTimer
  | StopTimer
  | NotifyServer Number
  | DismissNotificationAfter Int
  | ConsoleLog String

update :: Model -> Msg -> Tuple Model (Array Instr)
update model = case _ of
  Ticked deltaMs -> case model.timerState of
    Running { elapsedMs } ->
      let elapsed' = elapsedMs + deltaMs in
      if elapsed' >= toNumber model.durationMs
        then Tuple (model { timerState = Stopped }) [ StopTimer, NotifyServer elapsed' ]
        else Tuple (model { timerState = Running { elapsedMs: elapsed' } }) []
    Stopped -> Tuple model []
  -- …
Because the effects are data, asserting "this action fires StopTimer and NotifyServer 3100.0" is plain value equality — no mocks, no DOM, no runtime. The test doesn't even import Flame. 
it "tick reaching duration: Running => Stopped, fires StopTimer + NotifyServer" do
  let model = { timerState: Running { elapsedMs: 2900.0 }, durationMs: 3000, serverStatus: Idle }
  update model (Ticked 200.0)
    `shouldEqual` Tuple (model { timerState = Stopped }) [ StopTimer, NotifyServer 3100.0 ]
Naming note: we call the type Instr, not Cmd (Elm's Cmd is the opaque one — the opposite of this inspectable data) and not Effect (taken by PureScript's effect monad).

2 · The MVU loop, made testable

The pure core decides what to do; an impure interpreter at the edge decides how. The two are separate, which is exactly what keeps the logic trivially testable. 

flowchart LR
  V["view (DOM)"] -- "Msg" --> U["update (PURE)"]
  U -- "next model" --> V
  U -- "Array Instr (data)" --> I["interpret (Aff, at the edge)"]
  I -- "Msg" --> U
  classDef pure fill:#16241a,stroke:#4ade80,color:#e6e6e6;

Tests live entirely on the green box. The runtime, the DOM, and the Aff world all sit behind the orange box and never appear in a unit test.

3 · The interpreter — and recurring effects

interpret turns one Instr into an Aff (Maybe Msg). Most effects map to a single message: 

interpret :: Instr -> Aff (Maybe Msg)
interpret = case _ of
  NotifyServer _ -> do
    delay (Milliseconds 500.0)             -- fake a round-trip
    pure (Just (GotServerNotification (Right unit)))
  DismissNotificationAfter ms -> do
    delay (Milliseconds (toNumber ms))
    pure (Just DismissNotification)
  ConsoleLog s -> do log ("[Timer] " <> s); pure Nothing
  StartTimer -> do liftEffect startTickingImpl; pure Nothing
  StopTimer  -> do liftEffect stopTickingImpl;  pure Nothing
The subtle bit: a ticking timer emits many messages over time, but Aff (Maybe Msg) yields at most one. A recurring effect isn't a command — it's a subscription. So StartTimer/StopTimer just switch an interval on/off, and the ticks arrive through Flame's subscription channel: 
-- ticks are a subscription, not a command
subscriptions :: Array (Subscription Msg)
subscriptions = [ onCustomEvent' (EventType "tick") (Ticked 100.0) ]

This keeps interpret honest (Instr -> Aff (Maybe Msg), one message per effect) and removes the need for any "inject a message" back-channel.

4 · The glue: pure update → Flame update

Flame's update wants Array (Aff (Maybe Msg)); ours returns Array Instr. One generic adapter bridges them — and it's just a functor map over the second slot of the tuple (the model is passed through untouched): 

mkUpdate
  :: forall model msg instr
   . (instr -> Aff (Maybe msg))                       -- the interpreter
  -> (model -> msg -> Tuple model (Array instr))      -- the pure update
  -> (model -> msg -> Tuple model (Array (Aff (Maybe msg))))
mkUpdate interpret update model msg = rmap (map interpret) (update model msg)
--                                    ^^^^  Bifunctor rmap: touch only the cmds slot

5 · The time-travel debugger

Time-travel is fundamentally a runtime concern: to record you need a tap on the loop; to jump you must replace the model and re-render without tearing down the app. Flame's public API exposes neither — so we vendored Flame and added a tiny hook, the same way React/Vue devtools work: the runtime looks for an external hook on a global and, if present, streams transitions to it and hands it a control handle. No hook installed → behaviour unchanged.

  Hook -- "connect({ setModel, setPaused, dispatch })" --> RT
  Hook --> Panel["devtools.js panel
(timeline, diff, jump)"]
  Panel -- "setModel (jump) / setPaused (freeze)" --> RT

  class Hook,Panel ext;

The insight that made it small: setModel already existed

Flame's runtime already had a render that writes the model Ref and re-renders through the existing diff. That is a model setter — exposing it gives jumps for free, no re-mount (focus, scroll, listeners all survive). 

-- already in Flame's `run`, unchanged — we just expose it as `setModel`
render model = do
  rendering <- ER.read renderingState
  FRD.resume rendering (application.view model)   -- diff + patch the DOM
  ER.write model modelState                       -- update the model Ref

Pause = freeze without stopping the world

While scrubbing we set paused; runUpdate short-circuits, so subscriptions and external messages are simply dropped (the timer keeps firing tick events — the runtime ignores them). setModel bypasses runUpdate, so jumps still work while paused. 

runUpdate message = do
  isPaused <- ER.read pausedState
  unless isPaused do
    currentModel <- ER.read modelState
    let Tuple model affs = application.update currentModel message
    observeAll currentModel message model        -- record + notify devtools
    when (modelHasChanged currentModel model) (render model)
    -- … run affs …

The full patch (≈one module) is documented in FLAME_DEVTOOLS_PATCH.md, written so it could become an upstream proposal.

6 · Decisions & trade-offs

DecisionWhy
Effects as Instr data, interpreter at the edge Makes update pure → tests assert exact (model, [Instr]), no mocks. The whole point.
Recurring tick as a subscription, not a command Aff (Maybe Msg) is one-shot; a stream of ticks isn't an effect, it's a source.
Debugger via a runtime patch, not a userland wrapper Jumping needs in-place setModel + re-render; only the runtime can do that without a re-mount.
External global hook (React/Vue style), Foreign boundary Fully decouples the debugger from the app (could be a browser extension). Cost: model/msg cross as opaque values, pretty-printed heuristically.
Pause by dropping messages at runUpdate Generic freeze — no per-effect knowledge needed (timers, sockets, …). setModel still works while paused.
Collapse identical consecutive messages (default), raw-consecutive Tames Ticked spam. A different (even hidden) message breaks a run, so the timeline stays truthful.
One self-contained app per task (own vendor/flame) Tasks stay independent and buildable/testable one at a time. Trade: vendored Flame is duplicated (future: hoist to a workspace).
